OOM & TLE — how they show up in interviews
What OOM / TLE mean in interviews
In a live coding session, OOM and TLE are not error messages — they are follow-up prompts. After you solve a problem, interviewers routinely say: "What if the input is 10 billion rows?" "What if memory is limited to 2 KB?" "Your solution is O(n²) — can you do better?" This is where most candidates lose points.
The standard complexity thresholds
n ≤ 10⁶ → O(n log n) or O(n) expected
n ≤ 10⁵ → O(n log² n) may pass
n ≤ 10⁴ → O(n²) acceptable
n ≤ 10³ → O(n³) fine for small inputs
Python is ~3–5× slower than Java/C++ — factor in when choosing language
Google — memory / timeout focus
Signature follow-up: "Intervals are coming as a stream — you can't sort the full array." Forces a TreeMap / sorted-insert solution instead of sort + merge.

Stack overflow: Recursive DFS on a graph with millions of nodes is probed — the expected answer is iterative DFS with an explicit stack, or BFS.

Memory framing: "Sort 9 TB with 2 KB RAM" — classic external sort / streaming framing. Assesses distributed thinking.
Meta — memory / timeout focus
Stack overflow as DP follow-up: Recursive solution accepted first; then "can you avoid recursion?" — expected: bottom-up DP or iterative with explicit stack.

Graph deep-copy: Clone Graph (LC 133) starts recursive — follow-up probes risk of stack overflow on massive graphs → BFS/iterative DFS.

Cache design: LRU/LFU is inherently a memory management problem — O(1) access + O(1) eviction is the direct OOM proxy.
Apple — memory / timeout focus
Most explicit OOM culture: Apple explicitly tests "memory behavior on large inputs" — interviewers ask about heap vs. stack allocation, memory leaks, and reference counting in follow-ups.

On-device constraints: System design questions for iOS teams include battery budget, bandwidth, and memory footprint — OOM is a first-class constraint, not an afterthought.

Concurrency + memory: Race conditions and safe memory sharing are probed for OS-level teams.
TikTok / ByteDance — memory / timeout focus
TLE built into the OA: The HackerRank OA is designed so that O(n²) solutions hit TLE on large test cases. Candidates must optimize to O(n log n) or O(n) to pass all cases.

Recommendation system: Senior roles discuss petabyte-scale pipelines with millisecond SLAs — implicit timeout and memory constraints drive every design decision.

Complete binary tree: Tests awareness that random traversal wastes memory — must exploit structural properties to avoid brute force.
